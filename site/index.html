Evaluating Loyd's _detect_intent classifier

A small evaluation pipeline for the top-of-funnel intent classifier in Loyd's scheduling agent. The dataset is 25 deliberately-tricky emails; the eval reports where the classifier struggles, why, and what to do about it.

The eval pins the classifier's failures to a single mode, every email it got wrong, it got wrong by predicting none, traces it to a prompt that suppressed the model's reasoning, and the harness then validates the fix: a reasoning-first prompt lifts clean accuracy from 11/13 to 13/13. Cross-checks on five independent runs corroborate the failure is real, not a same-model artefact.

What the metric names mean (click to open)
Clean accuracy
Accuracy on just the 13 "clean" rows (one obvious correct label per email). The strict score, before any credit for either-way answers on ambiguous rows.
F1
The harmonic mean of precision (when the model predicts label X, how often is it actually X?) and recall (of all the actual X emails, how many did the model catch?). Between 0 and 1; high only when both precision and recall are high.
Macro-F1
The F1 for each of the 6 labels, averaged with equal weight, so the rare classes count as much as the common ones. The alternative, micro-F1, would let none (7 examples) dominate cancel (1 example); macro keeps the imbalanced label space honest. Caveat: with cancel at n=1, macro is high-variance, one prediction swings the score, so the report shows it next to the per-class breakdown.
ECE, Expected Calibration Error
The gap between what the model says it knows and what it actually gets right. Bucket predictions by confidence (0.5-0.7, 0.7-0.9, etc.); in each bucket compare mean confidence to actual accuracy; weight by bucket size. Lower is better. 0.0 = perfect calibration. Unstable at n=25, so it's reported alongside the reliability bins.
>0.95 auto-send precision
Of the predictions the model made at >95% confidence, what fraction were correct. This is the slice Loyd's pipeline auto-sends with no human review, so it's the single most decision-relevant number. The "auto-send n" is how many predictions landed in that band.
HITL
Human In The Loop. Loyd's Approval Gate: low-confidence predictions get routed to the user to review the draft before send. The routing threshold is what decides which ones.
label_fit (clean / ambiguous / misfit)
My 3-tier annotation of how well the 6-label space fits each email. Clean = one obvious label (13 emails). Ambiguous = two defensible labels, both recorded (7 emails). Misfit = no label in the 6 fits (5 emails), the model is forced to pick the least-wrong one. Almost half the dataset isn't clean, which is the point, it's calibration material, not a leaderboard set.
Acceptable-set accuracy
For ambiguous rows, the model is credited for either the gold label OR the documented alternate. Sits next to strict accuracy in the report, a model that always picks the alternate isn't penalised for the ambiguity tier.

Design choices the reviewer should know up front

The brief said "simulate the LLM predictions however you'd like." What I picked, and why, so none of this is hidden in a footnote:

Gold labels
I labeled the 25 emails myself by reading each one. I used claude-opus-4-7 as a labelling assistant, it drafted candidate labels and the label_fit tier (clean / ambiguous / misfit); I reviewed and finalised every row. So the dataset has same-model exposure on the label side, which is the obvious risk and the reason for the cross-checks below.
Primary predictor (the "v0" run)
claude-opus-4-7 with a direct prompt: "classify this message, return ONLY a JSON object." Same model family as the labeler, on purpose, the closest mirror to Loyd's production setup (single LLM call with a confidence score) makes the eval methodology directly transferable.
The A/B fix tested
claude-opus-4-7 again, but with a reasoning-first prompt: "reason step by step, then give the JSON object as the last line." Same model, different prompt, which lifted clean accuracy from 11/13 to 13/13. Output in results/opus-reasoning/.
Cross-checks (to defuse the same-model concern)
Three cross-check runs: claude-sonnet-4-6 (within family, for a weaker-tier comparison), llama4-scout-17b (Meta, independent family), and qwen3.5-122b-a10b (Alibaba, independent family, no-think mode). The two independent-family runs, Llama-4 and Qwen, hit the same 0.846 clean accuracy as Opus and the same E14 -> none miss. The same-model labeler caveat is substantially answered, though the structural fix at scale (split labeler and predictor; multi-annotator human gold for the ambiguous tier) is still in the dataset-growth plan.

What the eval is about

Loyd is an AI scheduling assistant. Every inbound message hits _detect_intent first, a classifier that drops the message into one of six buckets (schedule, reschedule, cancel, query_agenda, block_calendar, none). It is the first domino: an error here cascades. The take-home asked us to design and implement a way to measure whether that classifier is doing a good job, and to surface where and why it fails.

I'm submitting two artefacts: a design document (the strategy) and an executable prototype (the code, the run, the written interpretation). Both live on this site, navigable from the top bar.

Reading order

  1. Design, the eval pipeline strategy. Covers all six required sections (eval strategy, scorer design, dataset growth, failure-mode taxonomy, production feedback loop, extension to the other LLM stages) plus assumptions and the label-space expansion question the brief invited.
  2. Results & Interpretation, the prototype outputs and what the eval told us: the headline finding, scorer-by-scorer numbers, calibration, the fix tested, and the cross-check that addresses the same-model caveat.
  3. Discoveries, findings from the runs, pulled out as standalone signal.
