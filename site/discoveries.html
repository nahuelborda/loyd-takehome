Discoveries

What the eval surfaced, pulled out as standalone signal, separate from the methodology in Design and the narrative in Results & Interpretation. Each finding links back to its receipt.

From the runs

1. Every error was the same error

On the v0 run (claude-opus-4-7, direct prompt), 11 of 13 clean emails were correct. The two misses (E14, E20) were both real requests collapsed to none. Not scattered noise, one coherent failure shape. That single observation drove the whole interpretation and the F9 ("context blindness") addition to the taxonomy.

2. The errors err conservative, not aggressive

Every wrong call dropped a real request, never invented one. For an HITL scheduling agent that's the safer direction: a missed request gets surfaced when the user replies asking about it; an invented meeting becomes a wrong-number outbound email and burns trust with a third party. The failure shape is structurally aligned with the agent's risk posture.

3. A reasoning-first prompt closed the gap, no training data needed

The v0 prompt closed with "Return ONLY a JSON object." That instruction suppressed the reasoning the model needed on the borderline cases. Changing the closer to "Reason step by step, then give the JSON object as the last line" lifted clean accuracy from 0.846 to 1.000 and macro-F1 from 0.817 to 0.917. A/B tested via run_eval.py --reasoning. A prompt-level fix beats a fine-tune at this stage.

Three out of four hit the exact same 0.846 clean accuracy. All four made the same E14 -> none miss. Same-family bias was the obvious objection to a Claude-labeled, Claude-predicted eval, three independent-family cross-checks substantially answer it.

5. But the per-email patterns diverged

Same aggregate accuracy hid real per-item disagreement. E07 only Opus gets right. E19 only Llama-4 gets right. E20 only Llama-4 + Qwen get right. The per-email x per-model grid shows the structure, aggregate metrics lie, the grid doesn't.

6. 48% of the dataset doesn't fit the 6-label space

Of 25 emails: 13 "clean" (one obviously correct label), 7 "ambiguous" (a reasonable person could pick either of two labels), 5 "misfits" (no label in the 6 fits). Almost half. An accuracy score computed against the 6-label gold for these rows measures the wrong thing, which is why the eval treats label_fit as a first-class signal and ambiguous rows get acceptable-set scoring.

7. F9 ("context blindness") emerged from reading the predictions

E11, E14, E19 all had thread_so_far AND a Re: subject line in the prompt. The predictor passes all of it to the model. The model still dropped to none. F1-F8 didn't have a code for "context was present and the model ignored it", so I added F9. The eval surfaced a gap in its own taxonomy.
